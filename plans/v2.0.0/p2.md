## P2.1 — Taint Tracking / Data Flow Analysis

**Status:** Currently all 275 detectors are substring/pattern matching on source text. Zero inter-procedural analysis.

**What it is:**
- Track how user-controlled data flows through assignments, function calls, and returns
- Build a def-use chain: `c.Query("dsn")` → `dsn` → `factory(dsn)` → `sql.Open(...)`
- Detect when sanitized data reaches a sink (true negative) vs. unsanitized (true positive)

**Approach:**
- Implement reaching-definitions analysis per function (intra-procedural)
- Build a call graph across functions within a file (inter-procedural, intra-file)
- Use tree-sitter's scope/declaration queries for variable resolution
- Start with a simple subset: track string-typed variables through ≤2 hops of assignments

**Why it matters:**
- Current detectors miss `encrypt(secret)` wrapped arguments
- Current detectors can't track across function boundaries: `getDSN()` → `sql.Open()`
- This is the single biggest gap between CodeHound and a real security analyzer

**Estimated effort:** 4-6 weeks for intra-procedural; 8-12 weeks for inter-procedural

**Checklist:**
- [x] Design a `TaintGraph` structure (variables → assignments → call sites) — fully implemented in `src/lang/go/detectors/cwe/taint/model.rs`
- [x] Implement `build_taint_graph(root: Node) -> TaintGraph` during fact extraction — implemented in `taint/graph_query/build.rs`
- [x] Add `taint_source` and `taint_sink` annotations to `GoUnitFacts` — fields present in `facts/types.rs`, populated by `extract_taint_facts`
- [x] Rewrite CWE-22/78/89/79/90/91 detectors to use taint paths instead of substring matching — all six have taint-gated implementation
- [x] Add inter-procedural call graph (resolve function calls within same file and across files via project-level merge)
- [x] Add sanitizer detection (`filepath.Clean`, `html.EscapeString`, prepared statements) — all classified as sanitizers in `taint/extract/classify.rs`

## P2.2 — Baseline / Ignore-Once Mechanism

**Status:** ✅ **COMPLETED** — Baseline save/load/filter and inline/file suppression are implemented.

**What it is:**
- First run on a legacy codebase produces thousands of findings
- User runs `codehound --baseline` to save all current findings as "acknowledged"
- Subsequent runs: only NEW findings (not in baseline) cause non-zero exit
- Inline suppression: `// codehound-ignore: CWE-22` on specific lines

**Approach:**
- Baseline file: JSON/HashMap of `rule_id → [file:line:fingerprint]` entries in `.codehound-baseline.json`
- `--baseline` flag saves current findings as baseline
- Default behavior: if baseline file exists, filter out acknowledged findings from output
- `--no-baseline` to ignore the baseline
- Inline/file comment parsing during scan: `// codehound-ignore: RULE_ID` and `// codehound-ignore-file`

**Why it matters:**
- Without this, adopting CodeHound on an existing codebase is impossible — noise drowns signal
- Industry standard: ESLint `--cache`, RuboCop `.rubocop_todo.yml`, Semgrep baseline

**Estimated effort:** 1-2 weeks

**Checklist:**
- [x] Define baseline file format (`.codehound-baseline.json`)
- [x] Implement `Baseline` struct with fingerprint + location indexing
- [x] Add `--baseline` CLI flag (saves current findings)
- [x] Add `--no-baseline` CLI flag (ignores baseline file)
- [x] Integrate baseline filtering into `AnalysisResult` (filter findings before reporting)
- [x] Implement `// codehound-ignore: RULE_ID` comment parsing
- [x] Add `// codehound-ignore-file` for entire-file suppression
- [x] Add tests for baseline save/load/filter workflows

## P2.3 — Incremental Analysis

**Status:** ✅ **Phase 1+2+3+4.1+4.2+4.3+5+6.1 COMPLETED** — `CacheStore` is implemented and wired into the scan pipeline. End-to-end measured ~27× speedup on `gopdfsuit` (139 files: 0.4s cold → 0.015s warm). CLI flags `--no-cache` / `--cache-dir` / `--rebuild-cache` / `--prune-cache` and `[codehound.cache]` config block (`enabled`, `path`, `max_size_mb`) are in place. Dependency-based transitive invalidation (Phase 4.3) is implemented. Size-based LRU pruning (Phase 6.2) is deferred.

**What it is:**
- Cache parsed ASTs and extracted facts to disk
- On re-run, only parse files whose mtime or content hash has changed
- Reuse cached findings for unchanged files

**Approach:**
- Cache directory: `.codehound-cache/` with per-file entries
- Each cache entry: file hash → (ParsedUnit, findings, timestamp)
- On scan: check cache, skip re-parse if hash matches
- Invalidate cache entries whose dependencies changed (imported files)

**Why it matters:**
- 10-100× faster for repeated scans on large codebases
- Without this, CI time grows linearly with repo size on every commit
- Industry standard: Clippy (incremental via rustc), ESLint `--cache`, Ruff caching

**Estimated effort:** 2-3 weeks. ~1 week elapsed.

**Checklist:**
- [x] Define cache format (file hash → serialized findings per file)
- [x] Implement `CacheStore` with read/write to `.codehound-cache/`
- [x] Add content hashing during file walk (SHA-256 of source)
- [x] Skip re-parse when cache hit
- [x] Handle cache invalidation when imports/dependencies change (dependency extraction + transitive invalidation implemented)
- [x] Add `--no-cache` CLI flag
- [x] Add `--cache-dir` CLI flag for custom cache location
- [x] Add `--rebuild-cache` CLI flag
- [x] Add `--prune-cache` CLI flag
- [x] Add `[codehound.cache]` config block (`enabled`, `path`, `max_size_mb`)
- [x] Schema + TOML template updated
- [x] Unit + integration tests in `tests/engine_cache.rs` (26 tests)
- [x] Size-based LRU pruning when cache exceeds `max_size_mb` (implemented via `evict_to_size()`)
