> **Prerequisite:** All P0/P1 items from Review2.md and architecture-enhancement-2 plans completed.

## Without These Features: Rating 8.5/10

## P2.1 — Taint Tracking / Data Flow Analysis

**Status:** Currently all 275 detectors are substring/pattern matching on source text. Zero inter-procedural analysis.

**What it is:**
- Track how user-controlled data flows through assignments, function calls, and returns
- Build a def-use chain: `c.Query("dsn")` → `dsn` → `factory(dsn)` → `sql.Open(...)`
- Detect when sanitized data reaches a sink (true negative) vs. unsanitized (true positive)

**Approach:**
- Implement reaching-definitions analysis per function (intra-procedural)
- Build a call graph across functions within a file (inter-procedural, intra-file)
- Use tree-sitter's scope/declaration queries for variable resolution
- Start with a simple subset: track string-typed variables through ≤2 hops of assignments

**Why it matters:**
- Current detectors miss `encrypt(secret)` wrapped arguments
- Current detectors can't track across function boundaries: `getDSN()` → `sql.Open()`
- This is the single biggest gap between SlopGuard and a real security analyzer

**Estimated effort:** 4-6 weeks for intra-procedural; 8-12 weeks for inter-procedural

**Checklist:**
- [ ] Design a `TaintGraph` structure (variables → assignments → call sites)
- [ ] Implement `build_taint_graph(root: Node) -> TaintGraph` during fact extraction
- [ ] Add `taint_source` and `taint_sink` annotations to `GoUnitFacts`
- [ ] Rewrite CWE-22/78/89/79/90/91 detectors to use taint paths instead of substring matching
- [ ] Add inter-procedural call graph (resolve function calls within same file)
- [ ] Add sanitizer detection (`filepath.Clean`, `html.EscapeString`, prepared statements)

## P2.2 — LSP / Daemon Mode

**Status:** Every `slopguard` invocation re-parses all files from scratch.

**What it is:**
- A persistent daemon process that holds parsed ASTs and fact caches in memory
- Editor integration via Language Server Protocol (LSP)
- On file save: re-parse the changed file only, invalidate affected caches, re-run relevant detectors

**Approach:**
- Implement a `Daemon` struct that holds a `HashMap<PathBuf, ParsedUnit>` cache
- Watch filesystem for changes (`inotify` or polling)
- On change: re-parse the file, diff findings, push diagnostics to the editor
- LSP server: handle `textDocument/didChange`, `textDocument/didSave`, publish `textDocument/publishDiagnostics`

**Why it matters:**
- 10-50× faster for editor integration (no cold start, no re-parsing unchanged files)
- Enables real-time feedback in VSCode, Neovim, etc.
- Critical for developer adoption — nobody runs linters from CLI-only anymore

**Estimated effort:** 3-4 weeks for daemon; 4-6 weeks for LSP integration

**Checklist:**
- [ ] Implement `Daemon` with in-memory parse cache (`HashMap<PathBuf, CachedUnit>`)
- [ ] Add file watcher (polling or inotify)
- [ ] Implement incremental re-analysis (changed file only)
- [ ] Implement LSP server (`tower-lsp` or manual JSON-RPC)
- [ ] Handle `textDocument/didOpen`, `didChange`, `didSave`, `didClose`
- [ ] Publish diagnostics via `textDocument/publishDiagnostics`
- [ ] Add `slopguard daemon` CLI subcommand
- [ ] Add VSCode extension manifest

## P2.3 — Fix-Application (AST Rewrites)

**Status:** `Finding::with_fix` stores a plain `String` suggestion. No AST transformation machinery exists.

**What it is:**
- Structured fix descriptions: "replace node at byte range X..Y with text Z"
- Automatic application: `slopguard --fix` rewrites source files
- Preview mode: `slopguard --fix --dry-run` shows diff without applying
- Modeled after `rustfix` (for Clippy) and ESLint's `--fix`

**Approach:**
- Define a `Fix` struct with `Replace { range: Range<usize>, replacement: String }` and `Insert`, `Delete` variants
- Detectors emit `Fix` instead of (or alongside) string suggestions
- `--fix` mode: collect all fixes, sort by byte position (descending to preserve offsets), apply to source
- Write modified source back to disk (or to stdout for `--dry-run`)

**Why it matters:**
- Current `with_fix` is documentation-only — users must manually apply suggestions
- Automated fixes are table-stakes for modern linters
- Enables CI auto-fix workflows

**Estimated effort:** 2-3 weeks

**Checklist:**
- [ ] Define `Fix` enum (`Replace`, `Insert`, `Delete`, `Multi`)
- [ ] Add `fixes: Vec<Fix>` to `Finding` (alongside or replacing `fix: Option<String>`)
- [ ] Update `emit.rs` to accept structured fixes
- [ ] Implement `apply_fixes(source: &str, fixes: &[Fix]) -> String`
- [ ] Add `--fix` and `--fix --dry-run` CLI flags
- [ ] Add tests for fix application (no overlapping fixes, correct byte offsets)
- [ ] Update 5-10 high-value detectors with real structured fixes

## P2.4 — Baseline / Ignore-Once Mechanism

**Status:** Only ad-hoc `source.contains("// nolint:...")` in `gin_framework.rs:55`. No global mechanism.

**What it is:**
- First run on a legacy codebase produces thousands of findings
- User runs `slopguard --baseline` to save all current findings as "acknowledged"
- Subsequent runs: only NEW findings (not in baseline) cause non-zero exit
- Inline suppression: `// slopguard-ignore: CWE-22` on specific lines

**Approach:**
- Baseline file: JSON/HashMap of `rule_id → [file:line]` entries
- `--baseline` flag saves current findings as baseline (writes to `.slopguard-baseline.json`)
- Default behavior: if baseline file exists, filter out acknowledged findings from output
- `--no-baseline` to ignore the baseline
- Inline comment parsing during scan: check for `// slopguard-ignore:` patterns

**Why it matters:**
- Without this, adopting SlopGuard on an existing codebase is impossible — noise drowns signal
- Industry standard: ESLint `--cache`, RuboCop `.rubocop_todo.yml`, Semgrep baseline

**Estimated effort:** 1-2 weeks

**Checklist:**
- [ ] Define baseline file format (`.slopguard-baseline.json`)
- [ ] Implement `Baseline` struct with `contains(rule_id, file, line) -> bool`
- [ ] Add `--baseline` CLI flag (saves current findings)
- [ ] Add `--no-baseline` CLI flag (ignores baseline file)
- [ ] Integrate baseline filtering into `AnalysisResult` (filter findings before reporting)
- [ ] Implement `// slopguard-ignore: RULE_ID` comment parsing
- [ ] Add `// slopguard-ignore-file` for entire-file suppression
- [ ] Add tests for baseline save/load/filter workflows

## P2.5 — Incremental Analysis

**Status:** Every `slopguard` invocation re-parses every file.

**What it is:**
- Cache parsed ASTs and extracted facts to disk (or daemon memory)
- On re-run, only parse files whose mtime or content hash has changed
- Reuse cached facts for unchanged files
- This is distinct from the daemon — it works for CLI invocations too

**Approach:**
- Cache directory: `.slopguard-cache/` with per-file entries
- Each cache entry: file hash → (ParsedUnit, findings, timestamp)
- On scan: check cache, skip re-parse if hash matches
- Invalidate cache entries whose dependencies changed (imported files)
- Integrate with the daemon for in-memory caching

**Why it matters:**
- 10-100× faster for repeated scans on large codebases
- Without this, CI time grows linearly with repo size on every commit
- Industry standard: Clippy (incremental via rustc), ESLint `--cache`, Ruff caching

**Estimated effort:** 2-3 weeks

**Checklist:**
- [ ] Define cache format (file hash → serialized findings per file)
- [ ] Implement `CacheStore` with read/write to `.slopguard-cache/`
- [ ] Add content hashing during file walk (SHA-256 of source)
- [ ] Skip re-parse when cache hit
- [ ] Handle cache invalidation when imports/dependencies change
- [ ] Add `--no-cache` CLI flag
- [ ] Add `--cache-dir` CLI flag for custom cache location
- [ ] Integration with daemon mode (share the same cache)

|:-----:|---------|-----------|

## Rating Without These Features

**8.5/10** — with all architecture-enhancement-2 plans applied.

The application would be a technically excellent v0.x static analyzer: fast, well-organized, correct within its design constraints. The gap to 9+ is entirely about _new capabilities_ (data flow, LSP, fixes, baseline, incremental), not about fixing what already exists.

A 9.5/10 rating requires at minimum:
- Taint tracking (P2.1) — the core analysis gap
- Baseline/ignore (P2.4) — the adoption blocker
- Incremental (P2.5) — the CI performance requirement

The remaining 0.5 to 10/10 would be LSP and fix-application.
