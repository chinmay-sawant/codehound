## P2.1 — Taint Tracking / Data Flow Analysis

**Status:** Currently all 275 detectors are substring/pattern matching on source text. Zero inter-procedural analysis.

**What it is:**
- Track how user-controlled data flows through assignments, function calls, and returns
- Build a def-use chain: `c.Query("dsn")` → `dsn` → `factory(dsn)` → `sql.Open(...)`
- Detect when sanitized data reaches a sink (true negative) vs. unsanitized (true positive)

**Approach:**
- Implement reaching-definitions analysis per function (intra-procedural)
- Build a call graph across functions within a file (inter-procedural, intra-file)
- Use tree-sitter's scope/declaration queries for variable resolution
- Start with a simple subset: track string-typed variables through ≤2 hops of assignments

**Why it matters:**
- Current detectors miss `encrypt(secret)` wrapped arguments
- Current detectors can't track across function boundaries: `getDSN()` → `sql.Open()`
- This is the single biggest gap between SlopGuard and a real security analyzer

**Estimated effort:** 4-6 weeks for intra-procedural; 8-12 weeks for inter-procedural

**Checklist:**
- [ ] Design a `TaintGraph` structure (variables → assignments → call sites)
- [ ] Implement `build_taint_graph(root: Node) -> TaintGraph` during fact extraction
- [ ] Add `taint_source` and `taint_sink` annotations to `GoUnitFacts`
- [ ] Rewrite CWE-22/78/89/79/90/91 detectors to use taint paths instead of substring matching
- [ ] Add inter-procedural call graph (resolve function calls within same file)
- [ ] Add sanitizer detection (`filepath.Clean`, `html.EscapeString`, prepared statements)

## P2.2 — Baseline / Ignore-Once Mechanism

**Status:** Only ad-hoc `source.contains("// nolint:...")` in `gin_framework.rs:55`. No global mechanism.

**What it is:**
- First run on a legacy codebase produces thousands of findings
- User runs `slopguard --baseline` to save all current findings as "acknowledged"
- Subsequent runs: only NEW findings (not in baseline) cause non-zero exit
- Inline suppression: `// slopguard-ignore: CWE-22` on specific lines

**Approach:**
- Baseline file: JSON/HashMap of `rule_id → [file:line]` entries
- `--baseline` flag saves current findings as baseline (writes to `.slopguard-baseline.json`)
- Default behavior: if baseline file exists, filter out acknowledged findings from output
- `--no-baseline` to ignore the baseline
- Inline comment parsing during scan: check for `// slopguard-ignore:` patterns

**Why it matters:**
- Without this, adopting SlopGuard on an existing codebase is impossible — noise drowns signal
- Industry standard: ESLint `--cache`, RuboCop `.rubocop_todo.yml`, Semgrep baseline

**Estimated effort:** 1-2 weeks

**Checklist:**
- [ ] Define baseline file format (`.slopguard-baseline.json`)
- [ ] Implement `Baseline` struct with `contains(rule_id, file, line) -> bool`
- [ ] Add `--baseline` CLI flag (saves current findings)
- [ ] Add `--no-baseline` CLI flag (ignores baseline file)
- [ ] Integrate baseline filtering into `AnalysisResult` (filter findings before reporting)
- [ ] Implement `// slopguard-ignore: RULE_ID` comment parsing
- [ ] Add `// slopguard-ignore-file` for entire-file suppression
- [ ] Add tests for baseline save/load/filter workflows

## P2.3 — Incremental Analysis

**Status:** Every `slopguard` invocation re-parses every file.

**What it is:**
- Cache parsed ASTs and extracted facts to disk
- On re-run, only parse files whose mtime or content hash has changed
- Reuse cached facts for unchanged files

**Approach:**
- Cache directory: `.slopguard-cache/` with per-file entries
- Each cache entry: file hash → (ParsedUnit, findings, timestamp)
- On scan: check cache, skip re-parse if hash matches
- Invalidate cache entries whose dependencies changed (imported files)

**Why it matters:**
- 10-100× faster for repeated scans on large codebases
- Without this, CI time grows linearly with repo size on every commit
- Industry standard: Clippy (incremental via rustc), ESLint `--cache`, Ruff caching

**Estimated effort:** 2-3 weeks

**Checklist:**
- [ ] Define cache format (file hash → serialized findings per file)
- [ ] Implement `CacheStore` with read/write to `.slopguard-cache/`
- [ ] Add content hashing during file walk (SHA-256 of source)
- [ ] Skip re-parse when cache hit
- [ ] Handle cache invalidation when imports/dependencies change
- [ ] Add `--no-cache` CLI flag
- [ ] Add `--cache-dir` CLI flag for custom cache location

## Missing From This P2 Plan (track separately)

These items are **not** the same as the big P2 capability gaps above, but they
should still be tracked because they affect architecture quality or operational
correctness.

### A. Source cache is modeled but not populated

**Status:** `AnalysisResult` exposes `source_cache`, and export/reporting paths
accept it, but `Analyzer::analyze_paths` currently returns an empty cache.

**Why it matters:**

- Export falls back to re-reading files from disk even though the architecture
  already implies "read once, reuse downstream".
- This weakens the otherwise clean separation between scan-time data capture
  and reporting/export.
- It also becomes a blocker for future incremental analysis
  because the result object is not carrying the source snapshot it claims to own.

**Checklist:**

- [ ] Return populated `source_cache` from the scan pipeline
- [ ] Decide whether cache ownership lives in `scan_entry`, `scan_entries_parallel`,
      or a dedicated scan artifact type
- [ ] Ensure exporters and future reporters use in-memory source before disk
- [ ] Add regression tests proving export does not depend on a second file read

### B. Structured finding identity should be first-class in outputs

**Status:** `Finding::fingerprint()` exists, and SARIF emits a partial
fingerprint, but there is no unified first-class fingerprint contract across
all output formats and future baseline workflows.

**Why it matters:**

- Baseline, deduplication, ignore-once, and CI diffing all need the same stable
  identity semantics.
- Right now the architectural concept exists, but it is not yet fully elevated
  into the product contract.

**Checklist:**

- [ ] Define a canonical finding identity format and version it
- [ ] Emit it in JSON output, SARIF, and export artifacts consistently
- [ ] Reuse the same identity in future baseline and suppression files
- [ ] Add tests to protect fingerprint stability across refactors

### C. Detector output model needs to evolve beyond message string

**Status:** Findings are still optimized for reporting, not for richer
post-processing workflows.

**Why it matters:**

- Even before taint tracking, the architecture would benefit from a richer
  structured payload for detector evidence, sink/source classification,
  confidence, and suppression metadata.
- This reduces future rewrites when taint or baseline features land.

**Checklist:**

- [ ] Decide the minimum structured evidence model to add to `Finding`
- [ ] Separate user-facing message text from machine-facing detector evidence
- [ ] Reserve fields for confidence / tags / suppression / remediation metadata
- [ ] Keep JSON/SARIF compatibility explicit while evolving the schema

### D. Rule-pack extensibility is still compile-time only

**Status:** The current architecture is clean for bundled rules, but adding new
rule families still requires shipping them in-tree and rebuilding the binary.

**Why it matters:**

- For a tool that already has a strong plugin-like internal structure, external
  rule-pack loading is the next architectural step after bundled maturity.
- This is not required for the current product stage, but it is a real missing
  item if the roadmap is evaluating long-term architecture headroom.

**Checklist:**

- [ ] Decide whether external rule packs are in scope at all
- [ ] If yes, define the boundary: metadata-only packs vs. compiled detectors
- [ ] Preserve current registry-driven maintainability for bundled rules
- [ ] Avoid weakening startup time or rule determinism

### E. Observability/diagnostic instrumentation is still developer-oriented

**Status:** The engine has good error isolation, but there is not yet a fuller
operational story around timings, scan summaries, cache hit rates, and rule
cost visibility.

**Why it matters:**

- Once incremental or larger monorepo workflows are added, performance
  debugging becomes a product need rather than just a development need.
- This is architecture work because it affects where measurement hooks live.

**Checklist:**

- [ ] Add structured timing per major pipeline phase
- [ ] Add optional per-rule or per-detector timing in debug mode
- [ ] Surface scan stats in machine-readable output
- [ ] Keep instrumentation low-overhead when disabled
